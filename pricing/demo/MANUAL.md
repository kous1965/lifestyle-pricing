# コントロールパネル(デモ版) 使い方マニュアル

擬似ダイナミックプライシングの判定ロジックを、実際にブラウザ上で操作しながら
確認できるデモ画面です。

- **公開URL**: Streamlit Community Cloud上のデプロイ済みアプリ(共有されたリンク)
- **対象**: クライアント店舗の実商品(名称・現売価)
- **注意**: 値下げ/値上げの根拠になる「売れ行き」データは合成(ダミー)です。詳しくは
  [注意事項](#注意事項合成データについて) を参照してください。

---

## 画面構成

画面は大きく3つのエリアに分かれています。

```
┌─────────────────────────────────────────────────────────┐
│ タイトル: 擬似ダイナミックプライシング — コントロールパネル(デモ版)      │
│ ℹ️ 案内バナー(実データ/合成データの説明)                          │
├───────────────┬─────────────────────────────────────────┤
│               │  📊 サマリー(6つの件数カード)                  │
│  サイドバー     │     値下げ候補 / 値上げ候補 / 下限到達 /        │
│  (左側)        │     承認待ち / 警告 / 対象外・据え置き           │
│               │                                           │
│  ・判定パラメータ │  📋 判定結果一覧(表)                          │
│  ・基準日       │     商品コード / 商品名 / シナリオ / アクション /   │
│  ・乱数シード    │     現在価格 / 提案価格 / 理由 / 商品ページリンク   │
│  ・実商品データ  │  📁 商品ごとの最低売価・最高売価(折りたたみ)          │
│    (再取得ボタン)│     CSV一括登録 / 画面での個別編集                  │
│               │  📄 テキストレポート(折りたたみ)                 │
│               │  📄 CSVレポート(折りたたみ・ダウンロード可)         │
└───────────────┴─────────────────────────────────────────┘
```

### サイドバー(左側)

| 項目 | 内容 |
|---|---|
| 基準日 | 「今日」がいつかを指定する日付。判定はこの日を基準に計算される |
| 値下げ | 値下げ判定の参照日数、値下げ率(%) |
| 値上げ | 急売れ判定の直近日数、ベースライン倍率、値上げ率(%) |
| ハンチング防止・共通ガード | クールダウン日数、累計変動上限(基準価格比)、新商品保護期間(日)。用語の意味は[こちら](#なぜクールダウンやハンチング防止が必要なの初めての方向け解説)を参照 |
| 端数処理 | 価格の丸め方(x980 = 心理価格 / floor10 / none) |
| 合成データの乱数シード | 実商品にどの売れ行きパターンを割り当てるかを変える数値 |
| 実商品データ | スナップショット取得日時の表示と、再取得ボタン |

### メインエリア(右側)

- **サマリーカード**: 現在のパラメータで判定した結果を、アクション別の件数で表示
- **商品ごとの最低売価・最高売価**: クリックで開く折りたたみ欄。商品ごとに
  「これより下には絶対に下げない」「これより上には絶対に上げない」価格を
  設定できる(詳しくは下記「商品ごとの最低売価・最高売価を設定する」参照)
- **判定結果一覧表**: 商品1件ごとの判定結果。列の意味は下記「表の見方」を参照
- **テキストレポート/CSVレポート**: クリックで開く折りたたみ欄。ダウンロードボタンあり

## 基本的な使い方

1. **画面を開く** — 共有されたURLにアクセスするだけ。ログイン不要
2. **サイドバーのパラメータを動かす** — スライダーやセレクトボックスを操作すると、
   画面右側のサマリー・表・レポートが**その場で自動的に再計算**される
   (「実行」ボタンなどは不要)
3. **結果を確認する** — 表の「アクション」列で、その商品が値下げ候補になったか、
   値上げ候補か、警告扱いか等を確認する。「理由」列に判定根拠が一言で出る
4. **レポートを保存したい場合** — 画面下部の「テキストレポート」または
   「CSVレポート」を開き、ダウンロードボタンで保存する
5. **商品ラインナップ・実売価を最新化したい場合** — サイドバー下部の
   「クライアント店舗から実商品名・実売価を再取得(60件)」ボタンを押す
   (店舗の公開ページから商品名と現売価を60件取り直す。1〜2分程度かかる)

### パラメータをいじって試してみる例

- 「値下げ判定の参照日数」を短くする → より短期間で売れていない商品も
  値下げ候補に上がりやすくなる
- 「クールダウン日数」を長くする → 一度価格を変えた商品が、しばらく
  再判定の対象から外れる期間が伸びる
- 「累計変動上限」を小さくする → 定価から離れた提案が「承認待ち」に
  回されやすくなる(自動反映されない安全弁)
- 「合成データの乱数シード」を変える → 同じ実商品リストのまま、どの商品が
  値下げ候補/値上げ候補/警告になるかの組み合わせが変わる(デモのバリエーション用)

## 商品ごとの最低売価・最高売価を設定する

「値下げ率」などのパラメータはすべての商品に共通で効きますが、
「この商品だけは絶対に○○円より安くしない/高くしない」という**商品ごと(個別)の
指定**もできます。判定パラメータのすぐ下にある「商品ごとの最低売価・最高売価(任意設定)」
を開くと操作できます。

- **最低売価を設定すると**: 原価から自動計算される下限より、こちらの方が
  高ければこちらが優先されます。自動値下げは、この価格に達したら
  「下限到達」として止まり、それより安くはなりません
- **最高売価を設定すると**: 定価(基準価格)より、こちらの方が低ければ
  こちらが優先されます。自動値上げは、この価格を超えません
- 空欄のままなら、今まで通り原価・定価からの自動計算だけが使われます

### 設定方法は2通り

1. **画面で1件ずつ編集**(日常のちょっとした調整向け)
   表の「最低売価」「最高売価」の欄を直接クリックして数値を入力するだけ。
   入力すると自動的に保存され、次にこの画面を開いたときも値は残っています
2. **CSVで一括登録**(新商品をまとめて追加する時など)
   - 「現在の設定をCSVでダウンロード」で、今の商品一覧(商品コード・商品名・
     現在の設定)をCSVとして保存できます。それをExcel等で編集し、
     「最低売価」「最高売価」の列を埋めて保存し直してください
   - 編集したCSVを「CSVをアップロード」から読み込むと、内容がまとめて反映されます
   - CSVの列は「商品コード」「最低売価」「最高売価」の3つ(「商品名」列は
     見やすさのためだけで、読み込み時には使いません)。空欄の列はその項目を
     未設定のままにします

### 具体例

> ある商品は原価計算だと1,200円まで値下げできる計算になるが、
> 過去の経験から「1,500円より安く売ると採算が合わない」と分かっている場合、
> その商品の「最低売価」に1,500円と入力しておけば、自動値下げは
> 1,500円で必ず止まります(1,200円までは下がりません)。

## なぜ「クールダウン」や「ハンチング防止」が必要なの?(初めての方向け解説)

サイドバーに出てくる「ハンチング防止」「クールダウン」という言葉、最初は
分かりにくいと思うので、かみ砕いて説明します。

### そもそも「ハンチング」って何?

「ハンチング」というのは、値段が短い期間で
**「上げ→下げ→上げ→下げ…」と忙しく行ったり来たりしてしまう現象**のことです。

たとえるなら、調子の悪いエアコンが「暑い→冷やしすぎ→また暑い→また冷やしすぎ」を
繰り返して、ちょうどいい温度に落ち着かないようなイメージです。値段でも同じことが
起こり得ます。

**具体例:**

> 月曜日、ある商品がたまたま数件多く売れたので「値上げ」判定が出て
> 1,000円 → 1,050円に。
> ところが値上げした途端に売れ行きが落ち着いたので、火曜日には「値下げ」判定が出て
> 1,050円 → 980円に。
> さらに水曜日、また数件売れて「値上げ」判定…

これでは、
- お客さんが「昨日と値段が違う、この店は大丈夫かな?」と不安になる
- 確認する人(承認担当)も値段がコロコロ変わって収拾がつかなくなる
- そもそも「たまたま数件売れた/売れなかっただけ」の一時的なブレに、
  過剰に反応しすぎている可能性がある

という困りごとが起きます。これを防ぐ仕組みが「クールダウン」です。

### 「クールダウン」って何?

クールダウンは、**一度値段を変えたら、しばらくの間はまた自動で値段を
変えない「お休み期間」**を設ける仕組みです。

**具体例:**

> 「クールダウン日数」を7日に設定していたとします。
> 月曜日に980円へ値下げすると、そこから7日間(次の月曜日まで)は、
> たとえ「また値下げした方が良さそう」「今度は値上げした方が良さそう」という
> シグナルが出ても、自動では反映しません。7日経って落ち着いてから、
> 改めて判定します。

家計に例えると、「値引きシールを貼ったばかりのお惣菜に、その日のうちにまた
別の値引きシールを重ねて貼らない」ようなイメージです。少し様子を見てから、
次の対応を決める、という考え方です。

### 「方向転換ペナルティ」って何?(クールダウンの応用編)

クールダウン中でも、**直前と逆方向**(値下げした直後に値上げ判定が出た、など)の
シグナルが出た場合は、お休み期間を通常よりさらに伸ばします(既定では2倍)。
これが「方向転換ペナルティ」です。

**具体例:**

> 月曜日に値下げ(980円)。3日後の木曜日に「値上げ」のシグナルが出た。
> これは直前と真逆の動きなので、ハンチングの初期症状とみなし、
> クールダウンを7日間 → 14日間に延長。より慎重に様子を見ます。

### まとめ

| 言葉 | ひとことで言うと |
|---|---|
| ハンチング | 値段が上げ下げを繰り返して落ち着かなくなる、困った状態 |
| クールダウン | 値段を変えたあとの「お休み期間」。しばらく自動では動かさない |
| 方向転換ペナルティ | 逆方向のシグナルが出たら、お休み期間をさらに伸ばして慎重になる仕組み |

どれも、**「値段をコロコロ変えて、お客さんや現場を混乱させないための
安全装置」**だと思っていただければOKです。サイドバーの「クールダウン日数」
「累計変動上限」のスライダーは、このお休み期間や安全装置の“強さ”を
その場で調整できるようにしたものです。

## 表の見方(判定結果一覧)

| 列 | 意味 |
|---|---|
| 商品コード | 商品を一意に識別するコード |
| 商品名 | 実際の商品名(クライアント店舗から取得) |
| シナリオ(合成データ) | この商品にどんな売れ行きパターンを合成で割り当てたかのラベル(下記参照) |
| アクション | 判定結果。下表「アクションの意味」参照 |
| 現在価格 | 現在の売価(実データ) |
| 提案価格 | 判定ロジックが提案する価格(warn/skipなど提案がない場合は空欄) |
| 理由 | なぜそのアクションになったかの一言説明 |
| 商品ページ | 実際の商品ページへのリンク |

### アクションの意味

| アクション | 意味 |
|---|---|
| 値下げ候補 | 直近ずっと売れていないため、値下げを提案 |
| 値上げ候補 | 急に売れ行きが伸びているため、値上げを提案 |
| 下限到達 | 値下げしたいが、原価ベースの下限価格に達しているため、それ以上は下げない |
| 承認待ち | 提案価格が定価から大きく離れるため、自動反映せず人の承認を必要とする |
| 警告 | 赤字/送料負けの疑い、または二重価格表示中など、価格変更の前に確認が必要な状態 |
| 対象外/据え置き | 除外フラグ・新商品保護期間・クールダウン中・シグナルなし等の理由でそのまま |

### シナリオ(合成データ)ラベルの意味

実商品に割り当てている「売れ行きの演技パターン」です。実際の売れ行きではなく、
判定ロジックの各分岐を一通り見せるために意図的に作られたものです。

- 値下げ候補(直近売れず) / 値下げ→下限張り付き / 値上げ候補(急売れ)
- 値上げだが承認待ち(基準価格から乖離大) / 値下げだが承認待ち(基準価格から乖離大)
- クールダウン中(直近価格変更あり) / 新商品保護期間中 / 除外フラグ対象
- 赤字/送料負け警告 / 二重価格表示 警告 / 据え置き(シグナルなし)
- 在庫切れ多く判定保留

## 注意事項(合成データについて)

- **商品名・現売価**: クライアント店舗の公開ページから取得した実データ
- **売れ行き(注文件数・在庫切れ日・過去の価格変更)**: 合成(ダミー)データ。
  実際の注文実績ではありません。Yahoo!ショッピング注文APIとの連携は
  まだ実装されていないため、現時点では実際の売れ行きに基づく判定はできません
- **URLの公開範囲**: このデモはStreamlit Community Cloud上の公開アプリのため、
  URLを知っている人であれば誰でも閲覧・操作できます。社外への共有範囲にはご注意ください
- 価格は**このデモ画面から実際のYahoo!ショッピング上に反映されることはありません**
  (レポートのみモードのため、閲覧・シミュレーション専用です)
- **商品ごとの最低売価・最高売価の保存について**: 通常はアプリを閉じてもブラウザを
  変えても設定は残りますが、アプリ側のコードを更新・再デプロイした際は保存領域が
  リセットされ、設定が消えてしまう場合があります。重要な設定は「現在の設定をCSVで
  ダウンロード」で手元にも保存しておくことをおすすめします
